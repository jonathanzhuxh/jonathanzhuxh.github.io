Jonathan Zhu Case Study
Case Study

Build the model.
Change the habit.

Inheriting a broken regulatory reporting process with no documentation, no SOP, and upstream data that couldn't be automated — then rebuilding it from first principles, during a COVID lockdown, with no IT support and no budget. The answer wasn't better tooling. It was a different relationship with time.

01 The Problem

Everything manual.
Nothing documented.

OD — overdraft facilities — and EL — entrusted loans — are two routine but regulation-heavy product lines. Both carry daily, monthly, and quarterly reporting obligations across two regulators: NFRA and PBOC, each with their own field definitions, granularity requirements, and submission windows.

When I inherited the function, the person who had run it was gone. There was no handover document. No SOP. No data dictionary. The data lived in a collection of manually maintained spreadsheets, and it wasn't always clear what the numbers actually represented.

The process hadn't been designed. It had been survived.

Every monthly close was a six-day scramble: locating source data, cleaning it by hand, reconciling it against other teams' summaries, and generating submission files through a chain of manual steps that changed slightly every cycle. Errors surfaced late. The checker had little to work with.

The temptation was to immediately reach for the low-code tools that had just started being rolled out — Alteryx was being pushed precisely because the tech department had been paralysed by COVID and couldn't deliver anything. But automating an undocumented process just produces faster undocumented errors. That wasn't the place to start.

02 The Methodology

Don't fix month-end.
Fix every day.

The root problem wasn't the volume of work at month-end. It was that all the work — and all the errors — had been allowed to accumulate there. Data cleaning isn't something you do before a deadline. It's something you do continuously, so that by the time the deadline arrives, there's nothing left to clean.

A data model stays clean for the same reason a desk stays tidy: not because of a big clear-out, but because nothing is left to sit there overnight.

This shaped every design decision in the project — what to build first, how to structure the daily workflow, and why the automation at the end was fast to build and reliable from day one.

1 Map the data before touching the tools ▼
The first weeks produced only a document, not a workflow. OD and EL data had accumulated across dozens of manually maintained spreadsheets, with no shared structure and no data dictionary. Before any tooling, the job was to map what the data actually represented: five layers — corporate customer, loan contract, loan note, loan movement, guarantee contract — and to establish that every regulatory report was a different view of the same underlying structure, not a different structure.
Inherited processes carry inherited assumptions. The first deliverable is a map, not a fix.
2 Build the data model as a single source of truth ▼
With the map in place, the data model was built in Excel — five structured layers that both OD and EL data wrote into and all downstream reports read from. The model didn't know about regulators or submission formats. It only knew about the business: customers, contracts, notes, movements, guarantees. Regulatory logic lived in the processing layer, not the model. When reporting requirements changed, only the processing layer needed updating.
A data model is the answer to one question: what is the data actually about? Everything else is derived from that answer.
3 Close the data every day, not every month ▼
OD generates activity every working day. The daily workflow ran the same steps each morning: download the source file, process it through the Alteryx workflow, validate against the data model, sync. Any discrepancy surfaced the same day — not thirty days later, when the original transaction was harder to trace and the deadline was already close. By the time the monthly trigger fired, the data had been cleaned incrementally across the entire month. There was nothing left to reconcile.
Daily processing doesn't add work. It redistributes it — away from the deadline and toward the days when there's time to fix things properly.
4 Automate only what is already clean ▼
Monthly and quarterly automation was built last, after the daily workflow was stable and the data model was consistently clean. The monthly trigger reads from accumulated daily data, generates the submission file and working paper, and exits. It has no error-handling logic for data quality problems — because by the time it runs, those problems have already been resolved. Automating a clean process is simple. Automating a dirty process just produces faster dirty output.
The monthly close takes thirty minutes because three weeks of work happened before it. The automation is the easy part.

Manual entry (EL constraint)

EL data entry remained manual throughout — not by design, but by constraint. The upstream system for entrusted loans had no machine-readable export. Rather than block the automation on a dependency that couldn't be controlled, the data model absorbed the manual input cleanly: EL data entered once, in the right structure, propagated automatically to all downstream reports.

04 Key Decisions

Every constraint was
also a choice

The most important decisions weren't about tooling. They were about sequencing — what to do in what order, and what to leave undone.

The question
Map before building
The first instinct when inheriting a broken process is to start fixing. Instead, the first two weeks produced only a document: a complete map of the data model and the reporting requirements. Nothing was built. That map made everything built afterwards faster and more durable.
The question
Daily accumulation over month-end batch
Batching everything to month-end was the default because it felt like less work. It wasn't — it compressed all the data problems into the same 48-hour window as the submission deadline. Daily processing distributes the risk across the whole month.
The question
Accept EL manual entry
Automating EL source data would have required IT involvement that wasn't available. Rather than block on it, the architecture treated manual EL entry as a valid input layer — structured, versioned, and consumed automatically by all downstream workflows.
The question
Low-code over scripting
Alteryx was chosen not because it was the most powerful tool, but because the workflows it produced could be read and eventually maintained by colleagues without programming backgrounds. A system that runs without its creator is more valuable than a system only its creator can run.
05 The Result

Metrics as evidence,
not headline

The headline number is a ninety-six percent reduction in processing time at month-end. But that number is a consequence, not the achievement. The achievement is that data problems stopped arriving at month-end at all — because the daily workflow had already resolved them across the preceding thirty working days.

The checker's workload changed character as well as volume. Instead of hunting for errors in a six-day output, they were reviewing a working paper generated from a data model that had already been validated incrementally. The review became confirmation rather than discovery.

The deeper outcome is harder to quantify: a data model that stays clean indefinitely, because the process that feeds it treats daily hygiene as the job, not month-end recovery. Sustainability isn't a property of the system. It's a property of the daily habit the system enforces.

The problem wasn't that the process was manual. It was that nobody had decided what the process was supposed to be.
06 What Came Next

Automation without
transfer is a dependency

The handover on departure made one limitation visible. Colleagues who inherited the system could run it fluently — but when something broke or needed adapting, the automation was a black box to them. The system had been built to outlast its creator. It hadn't been built to be understood by them.

That gap — between a system that runs and a team that can evolve it — became the question that shaped every project that followed. Sustainable digitalisation isn't just about building something robust. It's about building the capability around it: documentation that explains the why, not just the what; tooling choices that colleagues without technical backgrounds can read and maintain; handovers that transfer ownership, not just access.

A system only its creator can modify is not infrastructure. It's a single point of failure with a better-looking dashboard.

The next project was designed with that lesson built in from the start.
